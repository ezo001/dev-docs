Multi-product documentation platform

Developer documentation · Case study · Sanitized for public portfolio

Problem

A portfolio of enterprise AI and platform products each maintained its own documentation in Word on SharePoint. Authors struggled with version drift, broken links, and no unified search. The build artifact exceeded hosting limits when images were bundled locally.

My role

  • Information architecture for 13 product doc sets under one site
  • Docusaurus configuration, sidebar generation, and search indexing
  • Authoring standards (MDX checklist, media handling, review workflow)
  • Operational docs: authentication setup, deployment runbooks, release process

Approach

I designed a docs-as-code model: Markdown in Git as source of truth, automated build to Azure Static Web Apps, and optional DOCX export for stakeholders who still work in SharePoint.

Information architecture

  • One landing page with filterable product cards
  • Separate route base paths per product (/industrial-ai, /ix-substrate, etc.)
  • Auto-generated sidebars from folder structure to reduce manual maintenance
  • Local full-text search across all doc sets

Media strategy

To stay under the 500 MB Static Web App limit, images are served from Azure Blob Storage via a remark plugin that rewrites ./media/ paths at build time. Authors keep familiar relative paths in Markdown; the pipeline handles the rest.

Outcomes

  • 13 product areas published under one searchable site
  • Build size reduced enough to deploy reliably on Azure SWA
  • Authors can preview locally before merge; optional PR review gate
  • Section-level edit links for fast SME corrections

Writing excerpt (landing page copy)

For internal use only. Filter, search, or select a card to explore documentation.

The landing page orients users in one sentence: what the site is for, how to navigate (filter, search, cards), and the access boundary. No marketing fluff — task-first.

What I'd highlight in an interview

  • Balancing engineer self-service with governance (review flags, SSO, branch protection)
  • Choosing doc plugins vs. monorepo folders for scale
  • Documenting the platform for authors who don't use Git daily
